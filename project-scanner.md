# Project Scanner

Two modes. Output is always `.pipeline/project-context.md`. Do not edit any source files.

## Mode Detection
- Passed an `instructionPath` → **INSTRUCTION mode**
- No `instructionPath` passed → **SCAN mode**

---

## INSTRUCTION MODE

User has provided a hand-written file describing the project. Trust it as authoritative. Do not scan the codebase.

### Step 1: Read instruction file

```bash
cat {instructionPath}
```

### Step 2: Validate coverage

Check the file covers these topics. For any topic missing or vague, note it in `## Known Gaps`:
- Tech stack (framework, UI, ORM, auth, state, testing)
- File structure (routes, components, API, types)
- Code conventions (naming, exports, CSS, typing style)
- API response shape
- Auth pattern
- Existing routes or models (at least approximate)
- Environment variables

### Step 3: Write `.pipeline/project-context.md`

Copy the instruction file content as-is, prepended with:

```markdown
# Project Context

> Source: instruction file ({instructionPath}). Provided by developer — not auto-scanned.
> Missing topics noted in Known Gaps below.

{instruction file content here}

## Known Gaps
- {any topics missing from instruction file, or "none"}
```

### Done
Return: `"INSTRUCTION mode. Context loaded from {instructionPath}. Gaps: {N}. Written to .pipeline/project-context.md"`

## SCAN MODE

Auto-scan the codebase. Write findings to `.pipeline/project-context.md`.

### Step 1: Dependencies

```bash
cat package.json
```

Extract and categorise:
- **Framework** — next, nuxt, vite+react, etc. + version
- **UI libraries** — shadcn/ui, antd, mui, chakra, etc.
- **ORM / DB** — prisma, drizzle, mongoose, typeorm, etc.
- **Auth** — next-auth, clerk, lucia, custom JWT, etc.
- **State management** — zustand, redux, jotai, react-query, etc.
- **Form handling** — react-hook-form, formik, etc.
- **Validation** — zod, yup, valibot, etc.
- **i18n** — next-intl, react-i18next, etc.
- **Testing** — vitest, jest, playwright, cypress, etc.
- **Utilities** — date-fns, dayjs, lodash, etc.

### Step 2: File Structure

```bash
find . -type f \( -name "*.ts" -o -name "*.tsx" \) \
  -not -path "*/node_modules/*" \
  -not -path "*/.next/*" \
  -not -path "*/dist/*" \
  -not -path "*/.git/*" | sort | head -80
```

Identify:
- Router type (App Router vs Pages Router for Next.js)
- Existing route paths (from `src/app/` or `src/pages/` folder names)
- Where components live
- Where API routes live
- Where types/utils/hooks are kept
- Test file locations and naming pattern

### Step 3: Auth Setup

```bash
find . -name "middleware.ts" -o -name "middleware.js" 2>/dev/null | grep -v node_modules
find . -name "auth.ts" -o -name "auth.js" 2>/dev/null | grep -v node_modules | head -5
grep -rn "JWT\|jsonwebtoken\|next-auth\|clerk\|lucia" src --include="*.ts" --include="*.tsx" -l 2>/dev/null | head -5
```

Identify: auth strategy, protected route mechanism, token storage, role model.

### Step 4: Database Setup

```bash
cat prisma/schema.prisma 2>/dev/null || echo "NO_PRISMA"
find . -name "drizzle.config.*" 2>/dev/null | grep -v node_modules
grep -rn "mongoose.connect\|createPool\|pg.Client" src --include="*.ts" -l 2>/dev/null | head -3
```

Identify: ORM, existing model names, soft-delete pattern, audit column pattern.

### Step 5: Code Conventions

Sample 3–5 existing component files:

```bash
find src -name "*.tsx" -not -path "*/ui/*" -not -path "*test*" 2>/dev/null | head -5
```

Read those files. Detect: component style, import alias, prop typing, export style, CSS approach, `'use client'` patterns, error handling.

### Step 6: API Patterns

```bash
find src/app/api -name "route.ts" 2>/dev/null | head -10
find src/pages/api -name "*.ts" 2>/dev/null | head -10
```

Read 2–3 existing API route files. Detect: response shape, validation library, error status codes.

### Step 7: Environment Variables

```bash
cat .env.example 2>/dev/null || cat .env.local.example 2>/dev/null || echo "NO_ENV_EXAMPLE"
grep -rh "process\.env\." src --include="*.ts" --include="*.tsx" 2>/dev/null | \
  grep -oE "process\.env\.[A-Z_0-9]+" | sort -u
```

### Step 8: Write `.pipeline/project-context.md`

```markdown
# Project Context

> Auto-generated by project-scanner (SCAN mode). Used by architect, dev-executor, and reviewer.

## Tech Stack
- **Framework:** {detected}
- **UI:** {detected}
- **ORM:** {detected}
- **Auth:** {detected}
- **State:** {detected}
- **Forms:** {detected}
- **i18n:** {detected}
- **Tests:** {detected}

## File Structure
- Routes: {detected pattern}
- Components: {detected pattern}
- API: {detected pattern}
- Types: {detected}
- Tests: {detected}

## Existing Routes
- {list from scan}

## Existing Models
- {list from scan}

## Code Conventions
- {list from scan}

## Auth Pattern
- {detected}

## Environment Variables
- {list from scan}

## Known Gaps / Deviations from Stack Default
- {anything that differs from stack file conventions, or "none detected"}
```

Fill each section from scan findings. If no findings, write `none detected`.

### Done
Return: `"SCAN mode. Stack: {framework}. Routes: {N}. Models: {M}. Written to .pipeline/project-context.md"`
